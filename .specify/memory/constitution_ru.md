# Конституция проекта: Speech Transit Toolformer

## 1. Миссия проекта

Этот проект создает ассистента с приоритетом на голосовое взаимодействие и
использование инструментов (tool-using) для итогового проекта курса DLS Speech.

Ассистент должен принимать речевые запросы пользователя на русском или английском языке,
определять необходимость вызова инструмента, при необходимости формировать корректный структурированный
вызов инструмента, опционально выполнять этот инструмент и выдавать обоснованный, понятный человеку ответ.

Главная исследовательская цель - это сравнение текстовых и аудио пайплайнов вызова инструментов и
оценка надежности, с которой небольшая мультимодальная модель может генерировать корректные вызовы инструментов из речи.

## 3. Основной инструмент

Основным function tool является:

`transport.where_is_vehicle`

Инструмент отвечает на вопросы о текущем местоположении маршрута общественного транспорта.

Обязательные аргументы:
* `city` (город)
* `transport_type` (тип транспорта)
* `route_number` (номер маршрута)

Разрешенные типы транспорта:
* `tram` (трамвай)
* `trolleybus` (троллейбус)
* `bus` (автобус)

Пример корректного вызова инструмента:

```json
{
  "tool_call": {
    "name": "transport.where_is_vehicle",
    "arguments": {
      "city": "Moscow",
      "transport_type": "tram",
      "route_number": "7"
    }
  }
}
```

Инструмент может быть сначала реализован как mock-сервис.
Интеграция с реальным API опциональна и не должна блокировать основную экспериментальную работу.

## 4. Неукоснительные принципы

### 4.1 Оценка с приоритетом на речь

Проект должен оценивать поведение на основе аудио, а не только текста.

Финальный отчет должен включать как минимум:
* оценку вызова инструментов только по тексту
* оценку ASR (распознавания речи) на синтетическом аудио
* оценку перехода "аудио -> инструмент"
* сравнение результатов для текста и аудио

### 4.2 Только структурированные вызовы инструментов

Когда требуется инструмент, модель должна выдать ровно один структурированный JSON-вызов инструмента.

В режиме вызова инструмента не допускаются поясняющий текст, markdown, комментарии или лишние поля.

### 4.3 Обычный текст, если инструмент не нужен

Ассистент должен оставаться универсальным.

Если запрос пользователя не требует использования инструмента поиска транспорта,
ассистент должен ответить обычным текстом и не должен галлюцинировать вызов инструмента.

### 4.4 Оценка перед оптимизацией

Решения о тонкой настройке промптов и дообучении (fine-tuning) должны основываться на метриках.

Дообучение опционально и должно выполняться только если базовое prompting недостаточно эффективно.

### 4.5 Воспроизводимые датасеты

Синтетические текстовые и аудио-данные должны генерироваться через скрипты или ноутбуки,
которые можно перезапустить.

Каждый элемент датасета должен включать:
* `id`
* `language` (язык)
* `user_text` (текст пользователя)
* `needs_tool` (нужен ли инструмент)
* `expected_tool_call` (ожидаемый вызов)
* `audio_path`, если аудио существует
* опциональные метаданные: спикер, движок TTS, город, тип транспорта, маршрут

### 4.6 Четкое разделение ответственности

Код проекта должен быть организован в отдельные модули:
* генерация данных
* генерация аудио
* инференс модели
* парсинг и валидация инструментов
* выполнение инструментов
* оценка
* отчетность

Скрытая логика только внутри ноутбуков не допускается для основной оценки.

### 4.7 Строгая типизация с Pydantic
Все структуры данных для аргументов инструментов и результатов должны быть определены с использованием `pydantic.BaseModel`.
Это обеспечивает:
* Автоматическую генерацию JSON Schema для промптов.
* Надежную валидацию выводов модели.
* Понятные сообщения об ошибках при сбоях парсинга.

## 5. Обязательные пайплайны

Проект должен поддерживать и оценивать следующие пайплайны:

### Пайплайн A

Текстовый запрос → модель → вызов инструмента

Цель: оценка следования текстовым инструкциям и форматирования вызова инструмента.

### Пайплайн B

Аудио-запрос → модель → транскрипт ASR

Цель: оценка качества распознавания речи с использованием WER (Word Error Rate).

### Пайплайн C

Аудио-запрос → модель → транскрипт + вызов инструмента за один проход

Цель: оценка совместного понимания речи и вызова инструментов.

### Пайплайн D

Аудио-запрос → модель → транскрипт → модель → вызов инструмента

Цель: оценка каскадного подхода ASR плюс текстовый вызов инструментов.

## 6. Обязательные метрики

Проект должен предоставлять отчеты по:

### Метрики использования инструментов

* доля разбираемых вызовов инструментов (parsable rate)
* точность полного совпадения вызова инструмента (exact-match accuracy)
* precision (точность)
* recall (полнота)
* уровень ложных срабатываний (false alarm rate)
* точность (per-slot accuracy) для `city`, `transport_type` и `route_number`

### Метрики ASR

* общий WER
* WER по языкам, если используются и русский, и английский
* уровень ошибок распознавания номера маршрута
* уровень ошибок распознавания города

### Метрики сравнения

* разрыв между текстовыми и аудио-вызовами инструментов
* сравнение пайплайнов A, C и D
* краткий анализ ошибок с примерами неудач

## 7. Правила датасета

Синтетический датасет должен содержать:
* не менее 200 текстовых примеров
* 10–20% примеров без необходимости инструмента (no-tool)
* примеры на русском и английском языках, если ограничения модели явно не документированы
* номера маршрутов с цифрами и буквенными суффиксами, например: `5`, `10`, `55a`, `90a`
* варианты названий городов на русском и английском
* варианты типов транспорта на русском и английском

Тестовая выборка (test split) должна быть фиксированной и использоваться во всех пайплайнах.

Примеры из обучающей выборки не должны попадать в тестовую.

## 8. Правила промптинга

Системные промпты должны определять:
* когда инструмент должен вызываться
* когда инструмент не должен вызываться
* точную JSON-схему
* правила нормализации для городов, букв маршрутов и типов транспорта
* примеры корректного и некорректного использования инструмента

Изменения промптов должны версионироваться или логироваться во время экспериментов.

## 9. Правила парсинга

Парсер должен:
* извлекать JSON-вывод модели
* валидировать имя инструмента
* валидировать обязательные аргументы
* валидировать разрешенные типы транспорта
* приводить буквы маршрутов к нижнему регистру
* возвращать четкую ошибку парсинга при невалидных данных
* использовать модели Pydantic для валидации
* логировать сырой вывод модели, если парсинг не удался (для анализа ошибок)
* нормализовать строковые аргументы (нижний регистр, удаление пробелов) перед валидацией

Система может выполнить одну попытку восстановления (repair retry) после невалидного JSON,
но все повторные попытки должны логироваться.

## 10. Правила выполнения инструмента

Инструмент транспорта должен возвращать обоснованный структурированный вывод.

Пример результата инструмента:

```json
{
  "status": "ok",
  "city": "Moscow",
  "transport_type": "tram",
  "route_number": "7",
  "nearest_stop": "Palikha Street",
  "direction": "Belorussky railway station",
  "updated_seconds_ago": 42
}
```

Финальный ответ ассистента должен основываться на этом результате инструмента.

Модель не должна выдумывать местоположение транспорта, если инструмент возвращает ошибку или
статус "недоступно".

* Рабочий текстовый датасет (200+ примеров).
* Реализация mock-инструмента с детерминированными ответами.
* Скрипт оценки Пайплайна A с логированием.

Приоритет 2:
* генерация аудио
* оценка ASR
* пайплайны B, C, D

Приоритет 3:
* тонкая настройка промптов
* опциональное текстовое SFT (Supervised Fine-Tuning)
* опциональное аудио-условное SFT
* опциональная интеграция с реальным API транспорта

## 13. Критерии готовности (Definition of Done)

Проект считается завершенным, когда:

1. Пользователь может предоставить текстовый или аудио-запрос о местоположении транспорта.
2. Модель может сгенерировать корректный вызов инструмента при необходимости.
3. Бэкенд может распарсить и выполнить вызов инструмента.
4. Ассистент может выдать финальный обоснованный ответ.
5. Пайплайны A–D оценены на фиксированном тестовом наборе.
6. Метрики и случаи неудач представлены в финальном отчете.
